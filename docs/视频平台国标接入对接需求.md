# 视频平台国标接入对接需求

## 背景

我方需要将机器狗前置摄像头视频接入视频主站平台，供远端用户通过主站统一鉴权、点播和播放。

我方现场设备提供 RTSP 视频流，计划在现场网关 PC 上部署 WVP-PRO + ZLMediaKit，将 RTSP 视频转换为 GB/T 28181 接入贵方视频主站。

```text
机器狗 RTSP → 现场网关 PC(WVP-PRO + ZLMediaKit) → GB/T 28181 → 视频主站 → 远端播放
```

根据现有《视频主站接口服务文档》，HTTP API、设备申请、通道查询、开始直播等业务接口已经有说明；当前主要缺口是 **GB28181/SIP/RTP 接入参数、网络互通方式、实际 API 凭证和联调环境信息**。

## 一、现有接口文档已明确的信息

### 1.1 HTTP API 调用规则

接口文档已说明：

| 项目 | 文档内容 |
|------|----------|
| 传输协议 | HTTPS |
| 请求方式 | POST |
| URL 格式 | `https://[主机]/api/v[版本号]/[接口名称]` |
| 请求体 | JSON |
| 响应格式 | JSON，包含 `code`、`message`、`data` |

公共请求头：

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| `Content-Type` | 是 | `application/json` |
| `auth-source` | 是 | `appKey`，由视频主站预先分配 |
| `auth-nonce` | 是 | 请求唯一标识，随机数，使用 UUID 生成 |
| `auth-timestamp` | 是 | 毫秒时间戳，请求有效期 3 分钟 |
| `auth-signature` | 是 | 请求签名 |

签名规则：

```text
auth-signature = md5(appSecret + paramStr + auth-nonce + auth-timestamp)
```

其中：

- `appSecret` 由视频主站分配。
- `paramStr` 为所有业务请求参数按 key 字母升序排序后，按 `key1 + value1 + key2 + value2...` 拼接。
- `auth-nonce` 使用 UUID。
- `auth-timestamp` 使用毫秒时间戳。

### 1.2 设备申请接口

接口文档已提供 `/devMgr/deviceApply`：

| 项目 | 内容 |
|------|------|
| 接口 | `/devMgr/deviceApply` |
| 方法 | POST |
| 说明 | 申请接入新设备，分配设备编码 |
| 适用范围 | 与视频主站直接连接的设备，如 NVR、IPC、平台等 |

设备类型已知：

| deviceType | 含义 |
|------------|------|
| `118` | NVR |
| `132` | IPC |

我方建议先按网关/NVR 方式申请：

```json
{
  "deviceType": "118",
  "regionCode": "440100",
  "deviceTag": "robot-dog",
  "deviceName": "山猫M20机器狗视频网关",
  "channelNum": 1,
  "remark": "机器狗前置摄像头"
}
```

接口文档说明申请成功后会返回：

| 字段 | 用途 |
|------|------|
| `deviceId` | 视频主站内部设备 ID，后续查询状态/信息使用 |
| `deviceCode` | 设备国标编码，配置连接主站参数时使用 |
| `accessPassword` | 设备接入密码/SIP 注册认证密码 |
| `channelList[].channelId` | 视频通道 ID，调用 `/stream/start` 时使用 |
| `channelList[].channelCode` | GB28181 通道编码 |
| `channelList[].channelScode` | 通道代码 |

### 1.3 状态查询接口

接口文档中已有设备和通道状态查询，但接口名疑似存在拼写：

| 文档接口名 | 用途 | 备注 |
|------------|------|------|
| `/devMgr/deviceStaus` | 查询设备状态 | 文档写作 `Staus`，需确认是否应为 `Status` |
| `/devMgr/channelStaus` | 查询通道状态 | 文档写作 `Staus`，需确认是否应为 `Status` |

状态返回字段：

| 字段 | 说明 |
|------|------|
| `online` | `0` 不在线，`1` 在线 |
| `deviceId` | 设备 ID |
| `channelId` | 通道 ID |

### 1.4 开始直播接口

接口文档已提供 `/stream/start`：

| 项目 | 内容 |
|------|------|
| 接口 | `/stream/start` |
| 方法 | POST |
| 说明 | 根据视频通道 ID 开启视频点播 |
| 请求字段 | `channelId` |

请求示例：

```json
{
  "channelId": "<channelList[0].channelId>"
}
```

返回内容包括：

| 字段 | 说明 |
|------|------|
| `streamId` | 视频流 ID |
| `channelId` | 通道 ID |
| `channelName` | 通道名称 |
| `FLV` | FLV 播放地址 |
| `WS_FLV` | WS-FLV 播放地址 |
| `RTMP` | RTMP 播放地址 |
| `HLS` | HLS 播放地址 |
| `RTSP` | RTSP 播放地址，文档示例中可能为空 |
| `audioEnable` | 是否开启音频 |
| `sVideoWH` | 原分辨率 |
| `sVideoFRate` | 原帧率 |
| `sVideoCode` | 原视频编码 |

播放策略文档已说明：

| 项目 | 文档内容 |
|------|----------|
| 无人观看停止 | 视频流 1 分钟内没有客户端观看会自动停止直播 |
| RTMP 鉴权 | RTMP 地址仅 3 分钟有效，且播放 1 次后失效，播放期间保活 |
| FLV/HLS 鉴权 | FLV、HLS 等 HTTPS 地址仅 3 分钟有效，首次播放后与 IP 绑定 |
| 断线重连 | 设备或播放客户端网络异常断开后，应重新获取新的播放地址 |

## 二、仍需视频平台方提供的信息

下面这些信息在现有接口文档中没有明确写出，是 GB28181 实际联调必须补齐的内容。

### 2.1 GB28181/SIP 上级平台参数

| 参数 | 需要平台方提供/确认 | 用途 |
|------|----------------------|------|
| 视频主站 SIP IP/域名 | 是 | WVP-PRO 注册到上级平台 |
| 视频主站 SIP 端口 | 是 | SIP REGISTER/INVITE/BYE 信令端口 |
| SIP 传输协议 | 是 | UDP 还是 TCP |
| 视频主站平台国标 ID | 是 | WVP 配置上级平台 ID |
| 视频主站 SIP 域 | 是 | GB28181 domain |
| 注册有效期 | 是 | REGISTER expires |
| 心跳周期 | 是 | Keepalive 间隔 |
| 认证用户名规则 | 是 | 是否使用 `deviceCode` 作为 SIP username |
| 认证密码规则 | 是 | 是否使用 `accessPassword` 作为 SIP password |

### 2.2 设备接入方式确认

现有文档说明 `118=NVR`、`132=IPC`，但我方场景是“机器狗 RTSP 通过网关接入主站”，请平台方确认：

| 问题 | 需要确认 |
|------|----------|
| 设备类型 | 我方应使用 `118` NVR/平台网关方式，还是 `132` IPC 方式 |
| 通道数量 | 当前按 1 路通道申请是否符合平台要求 |
| 通道编码 | 是否必须使用 `/devMgr/deviceApply` 返回的 `channelCode` |
| 设备编码 | 是否必须使用返回的 `deviceCode` 进行 SIP 注册 |
| 接入密码 | 是否直接使用返回的 `accessPassword` 做 SIP 注册认证 |

### 2.3 RTP 媒体流参数

| 参数 | 需要平台方提供/确认 | 用途 |
|------|----------------------|------|
| RTP 传输模式 | 是 | UDP、TCP active、TCP passive |
| RTP 收流端口范围 | 是 | 防火墙放行和 WVP/ZLM 配置 |
| SSRC 规则 | 是 | 是否由主站 SDP 指定，是否必须严格匹配 |
| RTP payload type | 是 | H.264 PS 常见为 96，需以平台为准 |
| 封装格式 | 是 | 是否要求 PS over RTP |
| 视频编码 | 是 | 支持 H.264 / H.265 哪些格式 |
| 音频要求 | 是 | 无音频是否可接受，是否必须补静音音轨 |
| 码率限制 | 是 | 单路推荐码率或上限 |
| 分辨率/帧率限制 | 是 | 例如 720p/25fps、1080p/25fps |

### 2.4 网络互通和防火墙

| 方向 | 需要确认 | 用途 |
|------|----------|------|
| 我方网关 → 视频主站 SIP | SIP IP、端口、协议 | 注册、心跳、点播信令 |
| 视频主站 → 我方网关 SIP | 是否需要主站主动访问我方网关 | INVITE/BYE 等下行信令 |
| 我方网关 → 视频主站 RTP | RTP 端口范围和协议 | 发送媒体流 |
| 视频主站 → 我方网关 RTP | TCP passive 等模式是否需要 | 媒体连接建立 |
| 我方业务服务 → 视频主站 API | API Base URL、端口、白名单 | 调用 HTTP API |
| 白名单 | 双方公网 IP 或 VPN/专线地址 | 网络安全策略 |

请平台方确认采用哪种互通方式：

- 公网 IP 白名单
- VPN
- 专线
- 内网专线/VPC 对接
- 其他方式

## 四、建议联调流程

1. 平台方提供 API Base URL、`appKey`、`appSecret`。
2. 我方调用 `/devMgr/deviceApply` 申请 1 路设备/通道。
3. 平台方确认返回的 `deviceCode`、`accessPassword`、`channelId`、`channelCode` 可用于 GB28181 接入。
4. 平台方提供 SIP 平台参数和 RTP 收流要求。
5. 双方完成网络白名单/VPN/专线互通。
6. 我方部署 WVP-PRO + ZLMediaKit，并使用 `deviceCode`、`accessPassword` 注册主站。
7. 平台方确认设备在线、通道在线。
8. 我方调用 `/stream/start`，平台方下发 INVITE。
9. 我方网关向平台发送 RTP/PS 媒体流。
10. 使用平台返回的 FLV/HLS/WS-FLV/RTMP 地址播放机器狗实时画面。

## 五、验收标准

联调完成后，应满足：

1. 视频主站显示我方设备在线。
2. 视频主站能看到 1 路机器狗视频通道。
3. 调用 `/stream/start` 后，主站能触发 GB28181 点播。
4. 我方网关能收到 INVITE，并向主站发送 RTP/PS 媒体流。
5. 主站返回的 FLV/HLS/WS-FLV/RTMP 地址能正常播放机器狗实时画面。
6. 播放断开后，重新调用 `/stream/start` 能恢复播放。
